# GhostLine — Инструкции за инсталация

GhostLine се състои от **3 отделни части**, които се пускат на твоя VPS:

1. **server** — API + WebSocket сървър (мозъкът, говори с базата данни)
2. **operator** — конзола за теб/операторите (отделен сайт, например `support.твоя-домейн.com`)
3. **client** — самостоятелна страница само с чата, която клиентите отварят
   директно (например `chat.твоя-домейн.com`) — **не е плаващ бутон**, цяла
   страница е чатът.

Всичко е self-hosted. Няма Telegram, Signal или друга трета страна.

---

## 0. Предварителни изисквания

На VPS трябва да имаш:
- Docker + Docker Compose (`docker compose version`)
- Отворени портове 80/443 (за Nginx/Caddy reverse proxy, препоръчително)
- (По желание, но силно препоръчително) домейн + Let's Encrypt за HTTPS,
  защото WebSocket в продукция трябва да е `wss://`

---

## 1. Сваляне на проекта

Разархивирай проекта на сървъра, например в `/opt/ghostline`.

```bash
cd /opt
unzip ghostline.zip -d ghostline
cd ghostline
```

---

## 2. Конфигурация на сървъра (server/.env)

```bash
cp server/.env.example server/.env
nano server/.env
```

Задължително смени:

```env
DATABASE_URL="postgresql://ghostline:СИЛНА_ПАРОЛА@postgres:5432/ghostline"
JWT_SECRET="генерирай-дълъг-случаен-низ-тук"
PORT=4000
CORS_ORIGIN="https://chat.твоя-домейн.com,https://support.твоя-домейн.com"
```

Генериране на случаен JWT_SECRET:
```bash
openssl rand -hex 32
```

Също смени паролата на postgres в `docker-compose.yml` (полето
`POSTGRES_PASSWORD` и съответно в `DATABASE_URL` по-горе трябва да съвпадат).

### 2.1 (По желание) Telegram нотификации до админите

Когато клиент пише ново съобщение (докато никой оператор не е поел чата),
или стартира изцяло нов разговор, ботът ще изпрати съобщение в Telegram.
**Важно:** съобщенията са E2E криптирани — ботът не може да изпрати
*съдържанието*, само "някой пише, отвори конзолата".

1. Отвори Telegram, намери **@BotFather**, изпрати `/newbot`, следвай
   стъпките — получаваш токен от вида `123456:ABC-...`.
2. Сложи токена в `server/.env`:
   ```env
   TELEGRAM_BOT_TOKEN="123456:ABC-..."
   ```
3. Изпрати произволно съобщение на своя нов бот (или го добави в група).
4. Намери chat id-то си:
   ```bash
   curl "https://api.telegram.org/bot<ТВОЯ_ТОКЕН>/getUpdates"
   ```
   В отговора търси `"chat":{"id": 123456789, ...}`. За група, id-то е
   отрицателно число.
5. Сложи го в `server/.env` (за повече админи — разделени със запетая):
   ```env
   TELEGRAM_ADMIN_CHAT_IDS="123456789,987654321"
   ```

## 3. Стартиране на цялата система

```bash
docker compose up -d --build
```

Това вдига 4 контейнера:

| Услуга    | Порт (по подразбиране) | Описание                          |
|-----------|-------------------------|------------------------------------|
| postgres  | 5432                    | база данни (само криптиран текст)  |
| server    | 4000                    | API + WebSocket                    |
| operator  | 3000                    | конзола за оператори                |
| client    | 8080                    | самостоятелна страница с чата       |

Проверка:
```bash
docker compose logs -f server
curl http://localhost:4000/health   # трябва да върне {"ok":true}
```

## 4. Миграции на базата данни

При първо стартиране трябва да създадеш таблиците:

```bash
docker compose exec server npx prisma migrate deploy
```

Ако това е първото изпълнение и нямаш още миграции, генерирай ги:

```bash
docker compose exec server npx prisma migrate dev --name init
```

## 5. Създаване на първи оператор (твоят акаунт)

Това става в две стъпки — първо генерираш ключова двойка в браузъра, после
създаваш акаунта.

### 5.1 Генерирай публичен ключ

Отвори `https://support.твоя-домейн.com` (или `http://VPS-IP:3000` временно)
— страницата ще ти даде форма за логин. Преди да имаш акаунт, отвори
конзолата на браузъра (F12 → Console) и изпълни:

```js
const nacl = await import('https://esm.sh/tweetnacl');
const kp = nacl.box.keyPair();
console.log('PUBLIC:', btoa(String.fromCharCode(...kp.publicKey)));
console.log('SECRET:', btoa(String.fromCharCode(...kp.secretKey)));
```

Копирай извежданите стойности — ще ги ползваш веднага.

### 5.2 Създай оператора

```bash
docker compose exec server npx ts-node src/seed.ts support ТВОЯ_ПАРОЛА PUBLIC_KEY_ОТ_ГОРЕ
```

(`support` е потребителското име — можеш да го смениш)

### 5.3 Запиши ключовата двойка в браузъра

Преди да логнеш, в конзолата на браузъра (на `support.твоя-домейн.com`)
изпълни — **с PUBLIC и SECRET от стъпка 5.1**:

```js
localStorage.setItem('ghostline_operator_keypair', JSON.stringify({
  publicKey: "PUBLIC_KEY_ОТ_5.1",
  secretKey: "SECRET_KEY_ОТ_5.1"
}));
```

Сега логни се с `support` / паролата си — операторската конзола ще
дешифрира съобщенията с този ключ.

> ⚠️ **Пази SECRET ключа на сигурно място (например в мениджър на пароли).**
> Той е единственото нещо, което дешифрира разговорите. Ако го изгубиш,
> старите разговори остават нечетими завинаги. Ако имаш няколко оператора
> на различни устройства, всеки трябва да повтори тази стъпка със своя
> собствена ключова двойка — но тогава клиентите ще криптират само към
> ключа на оператора, който е "claim-нал" разговора.

## 6. Настройка на клиентската страница (chat.твоя-домейн.com)

Файлът `client/public/config.js` казва на страницата към кой сървър да се
свързва:

```js
window.GHOSTLINE_CONFIG = {
  serverUrl: 'https://api.твоя-домейн.com',  // адресът на server частта
  label: 'Website Visitor',                   // по желание
};
```

Редактирай го **преди** `docker compose up --build`, или ако вече е вдигнато:

```bash
nano client/public/config.js
docker compose restart client
```

(благодарение на volume mount-а в `docker-compose.yml`, не е нужен ребилд)

Сега `https://chat.твоя-домейн.com` е **изцяло чатът** — клиентът отваря
линка и веднага вижда екран за разговор. Можеш да му дадеш този линк
директно (бутон "Контакт" на сайта ти, в био, в обяви и т.н.).

## 7. Reverse proxy + HTTPS (препоръчително)

Пример с **Caddy** (автоматичен HTTPS, най-лесен вариант):

```caddyfile
api.твоя-домейн.com {
    reverse_proxy localhost:4000
}

support.твоя-домейн.com {
    reverse_proxy localhost:3000
}

chat.твоя-домейн.com {
    reverse_proxy localhost:8080
}
```

```bash
sudo apt install caddy
sudo systemctl restart caddy
```

След това смени в `server/.env`:
```env
CORS_ORIGIN="https://chat.твоя-домейн.com,https://support.твоя-домейн.com"
```

И в `client/public/config.js`:
```js
serverUrl: 'https://api.твоя-домейн.com',
```

И в `docker-compose.yml` за operator:
```yaml
environment:
  NEXT_PUBLIC_API_URL: https://api.твоя-домейн.com
```

Ребилд след промени:
```bash
docker compose up -d --build
```

## 7.5 Какво вижда клиентът

`https://chat.твоя-домейн.com` сега има 3 екрана:

1. **Вход / Регистрация** — клиентът си прави потребител (потребителско
   име + парола). Това е отделна система от operator акаунтите — клиентски
   акаунти се пазят в таблица `ClientAccount`.
2. **Избор на отдел** — "Поддръжка" / "Поръчки" / "Друго". Това се записва
   на разговора и операторите могат да филтрират опашката по него.
3. **Чат екран** — статус "оператор онлайн / няма оператор онлайн" се вижда
   в горния десен ъгъл през цялото време (актуализира се на живо при
   логин/logout на операторите).

При повторно отваряне на страницата (същ браузър), токенът остава в
localStorage — клиентът пропуска входа и директно вижда чата си
(сървърът намира последния му незатворен разговор).

> ⚠️ E2E ключовата двойка остава локална за браузъра (както при операторите).
> Ако клиентът влезе от ново устройство със същия акаунт, получава нов
> ключ — старите съобщения, криптирани към стария ключ, остават нечетими
> на новото устройство. Самият акаунт обаче работи навсякъде (опашка,
> отдел, метаданни).

---

## 8. Тест на целия флоу

1. Отвори `https://chat.твоя-домейн.com` (от телефон/инкогнито) →
   екран "Вход / Регистрация" → направи си тестов клиентски акаунт
   → избери отдел ("Поддръжка") → виждаш "🔵 свързахте се... изчакваме оператор"
2. Отвори `https://support.твоя-домейн.com` → логни се → видиш чакащия
   разговор с таг "Поддръжка" → натисни **Claim**
3. Ако имаш настроен Telegram бот — трябва да получиш съобщение "🔔 Нов разговор"
4. Изпрати съобщение от операторската конзола → трябва да се появи
   разшифровано в клиентската страница (и обратно)
5. Пробвай прикачен файл (📎) и от двете страни
6. Затвори таб на оператора → статус "оператор онлайн" при клиента трябва
   да стане "няма оператор онлайн"

Ако виждаш `[unable to decrypt]` — означава, че публичните ключове не са
се разменили правилно (обикновено защото оператора още не е claim-нал
разговора, или ключовата двойка в localStorage е различна от тази, с която
сървърът знае оператора).

## Бърза справка — структура

```
ghostline/
├── server/      → api.твоя-домейн.com (порт 4000)
├── operator/     → support.твоя-домейн.com (порт 3000)
├── client/       → chat.твоя-домейн.com (порт 8080) — самостоятелна чат страница
├── widget/        → (по желание) вграждащ бутон за други сайтове
└── docker-compose.yml
```

`widget/` остава наличен, ако по-късно решиш да добавиш и плаващ бутон
към друг сайт — но за основната конфигурация (отделна чат страница) той
не е необходим.
